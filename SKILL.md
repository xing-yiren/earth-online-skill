# 地球Online Skill

> 将日常任务与习惯管理，包装成“地球Online闯关”式的 Agent Skill

---

## 1. Skill 定位

地球Online是一个运行在 Agent 中的生活游戏化 Skill。

它通过对话理解、定时播报与状态记录，让用户像在“地球Online”中闯关一样推进每天的生活、工作和学习。

它的设计目标不是替代宿主平台的 Agent、Memory 或 Session 系统，而是作为一个可挂载在不同宿主平台之上的“游戏化生活层”。

V1 的核心不是完整生活操作系统，而是围绕“每日任务推进”建立一个最小可用闭环：

- 早安播报
- 创建任务
- 完成任务
- 结算积分
- 解锁成就
- 晚间结算
- 兑换奖励

---

## 2. 宿主平台边界

### 2.1 适配目标

本 Skill 设计为可被不同宿主平台加载，包括但不限于：

- qclaw
- openclaw
- jiuwenclaw

Skill 仓库由 GitHub 承载，宿主平台负责安装、加载和调用。

### 2.2 宿主平台职责

宿主平台负责：

- 用户身份与 profile
- 长期 memory
- 最近对话上下文
- session 管理
- 意图识别
- 工具调度
- 定时触发
- Skill 安装与运行时承载

### 2.3 地球Online Skill 职责

地球Online Skill 负责：

- 将用户日常映射为“地球Online”任务体验
- 维护任务、积分、成就和奖励等私有游戏状态
- 输出早安播报、完成反馈、晚间结算与奖励兑换结果
- 提供统一的业务规则与数据接口

### 2.4 私有状态边界

地球Online 只维护自身业务状态：

- tasks
- points
- achievements
- rewards

地球Online 不应成为用户长期记忆的主来源，也不应复制宿主平台已经维护的完整 memory。

---

## 3. Host Context 接入原则

### 3.1 主原则

Skill 优先读取宿主平台提供的统一上下文，而不是直接依赖某个平台专属的 memory 文件格式。

也就是说，Skill 不应该写死为：

- 只读取 `MEMORY.md`
- 只读取 `main.sqlite`
- 只读取 `.qclaw` 或 `.jiuwenclaw` 某固定路径

Skill 应依赖宿主传入的统一 `host_context`。

### 3.2 Host Context 最小结构

宿主调用 Skill 时，建议至少传入如下结构：

```json
{
  "host": {
    "platform": "qclaw"
  },
  "user": {
    "id": "demo-user"
  },
  "session": {
    "current_date": "2026-03-25"
  }
}
```

### 3.3 推荐结构

如果宿主平台能够提供更多上下文，建议传入：

```json
{
  "host": {
    "platform": "jiuwenclaw",
    "skill_runtime_version": "1.0"
  },
  "user": {
    "id": "demo-user",
    "name": "DemoUser",
    "timezone": "Asia/Shanghai"
  },
  "session": {
    "session_id": "sess_001",
    "current_date": "2026-03-25",
    "current_time": "07:10:00+08:00"
  },
  "intent": {
    "name": "create_task",
    "confidence": 0.93
  },
  "context": {
    "recent_messages": [
      "我今天要完成项目报告",
      "还有面试要准备"
    ],
    "memory_facts": [
      "最近在准备面试",
      "本周需要交项目报告"
    ],
    "preferences": [
      "喜欢游戏化表达",
      "不喜欢太正式的语气"
    ]
  }
}
```

### 3.4 降级策略

如果宿主平台暂时无法提供完整上下文，Skill 可以使用本地默认配置进行降级：

- `examples/seed-examples/seed-examples/seed-data/USER.md` 作为默认配置来源
- `examples/seed-examples/seed-examples/seed-data/MEMORY.md` 仅作为开发调试或兜底来源

但这两者都不应被定义为跨平台正式接入时的主依赖。

## 4. V1 目标

### 2.1 产品目标

V1 要解决的核心问题是：

1. 让用户每天有明确的“开局感”
2. 让用户完成任务时获得即时反馈
3. 让用户在每天结束时看到“结算感”
4. 让长期坚持被积分、等级、成就和奖励承接

### 2.2 成功标准

满足以下条件即可认为 V1 成立：

1. 用户能通过自然对话创建任务
2. 用户能通过自然对话完成任务
3. 完成任务后能立即获得积分反馈
4. 能输出早安播报与晚间结算
5. 连续使用数天后，用户能看到成就、等级和奖励累积

---

## 5. Skill 边界

### 3.1 V1 包含范围

- 主线任务管理
- 支线任务管理
- 积分与等级
- 成就解锁
- 奖励兑换
- 早安播报
- 每日结算

### 3.2 V1 不包含范围

以下内容明确排除在 V1 之外：

- 饮食、饮水、心情、消费等硬结算
- 排行榜、多用户、社交系统
- Web dashboard 与分享卡片
- FastAPI 服务依赖
- iPhone 快捷指令集成
- 外部 API 自动拉取天气、日历等信息
- 复杂惩罚机制
- 多货币系统（如幸福值、健康值、美食值）

## 6. 世界观与语气

### 4.1 世界观映射

- 用户 = 玩家
- 每天 = 今日副本
- 推进日常 = 闯关
- 晚上总结 = 每日结算
- 长期坚持 = 生存记录 / 成就进度

### 4.2 回复风格

Skill 的回复应符合以下原则：

- 有游戏感，但不过度夸张
- 有陪伴感，但不过度拟人
- 以鼓励为主，不高压训斥
- 默认使用“标准游戏化风格”

### 4.3 文案原则

- 平时表达尽量自然，不要每句都使用重度 RPG 语气
- 在关键节点强化仪式感，例如早安播报、成就解锁、晚间结算、奖励兑换
- 用户完成任务后，必须明确指出做成了什么、获得了什么

## 7. 核心触发意图

### 5.1 早安播报

用户可能说：

- 早安
- 开启今天
- 今日副本
- 今天有什么任务

Skill 应：

- 读取当前状态
- 输出今日开局信息
- 展示今日主线、支线、积分、等级与鼓励语

V1 中与“早起”相关的判定口径为：

- 早起不直接等于真实生理起床时间
- 早起定义为“用户在晨间签到窗口内完成一次有效晨间签到”
- 有效晨间签到可由以下行为触发：
  - 用户主动说“早安”
  - 用户主动说“开启今天”
  - 用户主动触发“今日副本”
  - 宿主传入明确的 morning check-in 事件

如果签到时间在配置的晨间窗口内，则记为一次早起成功，并可用于 `early_bird_streak` 计算。

### 5.2 创建任务

用户可能说：

- 我今天要完成项目报告
- 明天要准备面试
- 以后每天阅读30分钟
- 我想坚持晨跑

Skill 应：

- 判断这是“创建任务”意图
- 识别任务名称、任务类型、可选截止时间
- 调用任务创建工具
- 返回任务创建结果

### 5.3 完成任务

用户可能说：

- 我完成了项目报告
- 晨跑打卡了
- 阅读结束了

Skill 应：

- 判断这是“完成任务”意图
- 尝试匹配已有任务
- 调用任务完成工具
- 返回积分变化与可能解锁的成就

### 5.4 每日结算

用户可能说：

- 今天做了什么
- 每日结算
- 今天结算一下

Skill 应：

- 汇总今日任务推进情况
- 输出积分、等级、成就与未完成任务

### 5.5 奖励兑换

用户可能说：

- 我想兑换看电影
- 有什么奖励可以换
- 查看奖励列表
- 确认兑换

Skill 应：

- 查询奖励
- 检查积分是否足够
- 必要时进行二次确认
- 完成兑换并反馈结果

## 8. Agent 行为规则

### 6.1 语义理解优先

Skill 必须优先依赖 Agent 的语义理解能力，而不是只依赖关键词匹配。

在跨平台模式下，任务主识别由宿主 Agent 完成，Skill 负责业务规则裁定和状态更新。

### 6.2 低风险操作可直接执行

以下操作可直接执行：

- 查看今日任务
- 查看积分与等级
- 查看已解锁成就
- 查看奖励列表
- 查看每日结算

### 6.3 中风险操作建议确认

以下情况建议确认：

- 当完成任务匹配到多个候选时
- 当创建任务缺少关键信息时
- 当兑换奖励消耗积分较高时

### 6.4 支持纠错

如果用户表示：

- 不是这个意思
- 你理解错了
- 撤销上一条

Skill 应优先进入纠错流程，而不是继续推进错误状态。

## 9. V1 数据原则

### 7.1 单一真相源

- tasks：任务状态
- points：积分与等级
- achievements：成就与统计
- rewards：奖励目录与兑换历史

### 7.2 数据闭环

当用户完成任务时，必须按顺序完整执行：

1. 更新任务状态
2. 增加积分
3. 检查成就
4. 写入历史记录
5. 返回结构化结果给 Agent

### 7.3 职责划分

#### Agent 负责

- 理解自然语言
- 判断用户意图
- 生成世界观文案
- 调用结构化工具
- 从宿主 memory 中整理并传入 `host_context`

#### 脚本或工具层负责

- 读写数据
- 修改任务状态
- 计算积分与等级
- 检查成就
- 返回结构化结果

## 10. V1 功能要求

### 8.1 早安播报

必须包含：

- 用户称呼
- 今日日期
- 累计生存天数
- 连续记录（如连续早起）
- 今日主线任务
- 今日支线任务
- 当前积分与等级
- 一句鼓励语

### 8.2 任务系统

V1 只保留两类任务：

#### 主线任务

- 重要且明确的任务
- 通常有截止时间
- 示例：完成项目报告、准备面试、提交方案

#### 支线任务

- 日常习惯或轻量目标
- 可以是每日重复任务
- 示例：晨跑、阅读30分钟、背单词

说明：V1 不单独设计“长期任务”作为独立类型。长期坚持主要通过支线任务的持续记录与成就系统承接。

### 8.3 积分系统

推荐规则：

- 主线任务完成：50-100 分
- 支线任务完成：10-30 分
- 成就解锁：10-200 分

V1 以正反馈为主，默认不启用强扣分规则。

### 8.4 等级系统

等级只做轻量长期反馈，建议保留：

- Lv1 新手玩家
- Lv2 资深玩家
- Lv3 高级玩家
- Lv4 精英玩家
- Lv5 大师玩家
- Lv6 传奇玩家

### 8.5 成就系统

V1 仅保留三类成就：

#### 生存类

- first_day
- survivor_7
- survivor_30

#### 早起类

- early_bird_3
- early_bird_7
- early_bird_30

说明：
V1 的早起类成就基于“晨间签到窗口内成功签到”的连续记录，而不是用户真实起床时间。

#### 任务类

- task_master_10
- task_master_50

### 8.6 晚间结算

必须包含：

- 今日主线完成情况
- 今日支线完成情况
- 今日积分变化
- 今日新成就
- 当前等级与距离下一等级差值
- 明日可延续任务

### 8.7 奖励兑换

用户必须能够：

- 查看奖励列表
- 发起兑换
- 二次确认兑换
- 查看兑换结果
- 查看兑换历史

## 12. 验收标准

Skill 在 V1 阶段至少应满足以下场景：

1. 用户说“早安”，能收到完整今日开局信息
2. 用户说“我今天要完成项目报告”，能正确创建主线任务
3. 用户说“以后每天阅读30分钟”，能正确创建支线任务
4. 用户说“我完成了项目报告”，能正确更新任务状态并加分
5. 连续记录或累计任务数达到阈值时，能正确解锁成就
6. 用户说“今天做了什么”，能输出每日结算
7. 用户说“我想兑换看电影”，能正确检查积分、确认兑换并记录历史
