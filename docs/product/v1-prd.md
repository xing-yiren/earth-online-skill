# 地球Online Agent Skill - V1 PRD

> 将日常任务与习惯管理，包装成“地球Online闯关”式的 Agent Skill

---

## 1. 产品概述

### 1.1 产品名称
**地球Online**

### 1.2 一句话定义
地球Online是一个运行在 Agent 中的生活游戏化 Skill，通过对话理解、定时播报和状态记录，让用户像在“地球Online”里闯关一样推进每日生活、工作与学习。

### 1.3 v1 核心定位
v1 不是完整生活操作系统，不做全量生活记录，而是专注于一个最小可用闭环：

**早安播报 -> 创建任务 -> 完成任务 -> 获取积分/成就 -> 晚间结算 -> 兑换奖励**

### 1.4 目标价值
- 让用户每天有“开局感”
- 让完成任务这件事有即时正反馈
- 让每一天结束时有“结算感”
- 让坚持行为被积分、等级、成就和奖励承接

### 1.5 平台定位
地球Online 以 GitHub 仓库形式承载，定位为一个可被不同宿主平台加载的跨平台 Skill。

V1 目标兼容的宿主平台包括：
- qclaw
- openclaw
- jiuwenclaw

---

## 2. 平台接入策略

### 2.1 核心策略
地球Online 不自建主 Agent、主 Memory 或主 Session 系统，而是依赖宿主平台提供用户上下文和调用能力。

### 2.2 宿主平台职责
宿主平台负责：
- 用户身份与 profile
- 长期 memory
- 最近对话上下文
- session 管理
- 意图识别
- 工具调度
- 定时触发
- Skill 安装和加载

### 2.3 地球Online Skill 职责
地球Online Skill 负责：
- 将日常任务与习惯映射为游戏化体验
- 维护任务、积分、成就与奖励等私有状态
- 输出开局播报、完成反馈、每日结算与奖励兑换结果

### 2.4 私有状态边界
地球Online 只维护自身业务状态：
- tasks
- points
- achievements
- rewards

宿主平台已有的长期 memory 不应由地球Online 重复维护为另一套主数据源。

### 2.5 Host Context 接入原则
宿主平台调用地球Online时，应优先向 Skill 传入统一的 `host_context`，而不是要求 Skill 直接读取宿主私有目录、sqlite 或 markdown memory 文件。

#### 最小结构
```json
{
  "host": {
    "platform": "qclaw"
  },
  "user": {
    "id": "demo-user"
  },
  "session": {
    "current_date": "2026-03-25"
  }
}
```

#### 推荐结构
```json
{
  "host": {
    "platform": "jiuwenclaw",
    "skill_runtime_version": "1.0"
  },
  "user": {
    "id": "demo-user",
    "name": "DemoUser",
    "timezone": "Asia/Shanghai"
  },
  "session": {
    "session_id": "sess_001",
    "current_date": "2026-03-25",
    "current_time": "07:10:00+08:00"
  },
  "intent": {
    "name": "create_task",
    "confidence": 0.93
  },
  "context": {
    "recent_messages": [
      "我今天要完成项目报告"
    ],
    "memory_facts": [
      "本周需要交项目报告"
    ],
    "preferences": [
      "喜欢游戏化表达"
    ]
  }
}
```

### 2.6 降级策略
如果某宿主平台暂时无法传入完整 `host_context`，地球Online 可以使用本地默认配置作为降级方案，但不应把本地 `MEMORY.md` 视为跨平台正式接入的主依赖。

## 3. 目标用户

### 2.1 核心用户
- 喜欢游戏化表达的人
- 有轻量任务管理需求的人
- 想要 Agent 陪伴感的人
- 希望建立“每日开始 / 每日结算”仪式感的人

### 2.2 非目标用户
- 需要严肃项目管理/GTD的人
- 需要复杂报表和统计分析的人
- 需要高压监督与惩罚机制的人

## 5. 设计原则

### 4.1 Agent 优先
自然语言理解由 Agent 完成，脚本只负责结构化执行和状态更新。

在跨平台模式下，宿主 Agent 负责主语义识别，地球Online 负责游戏规则映射与状态流转。

### 4.2 正反馈优先
v1 以鼓励和反馈为主，不默认启用强惩罚。

### 4.3 轻量优先
用户不需要维护复杂数据，尽量通过对话自动理解。

### 4.4 仪式感优先
每日体验至少要有两个关键时刻：
- 早晨开局
- 晚间结算

## 6. 核心用户流程

### 5.1 每日主流程

#### 早晨开局
1. 用户主动说“早安”或定时触发
2. Skill 输出今日开局信息
3. 用户知道今天有哪些任务要推进

#### 白天推进
1. 用户在对话中表达今天要做的事
2. Agent 识别并创建任务
3. 用户完成后在对话中打卡
4. Skill 更新任务状态并奖励积分

#### 晚间结算
1. 用户主动说“今天做了什么”或定时触发
2. Skill 汇总今日任务推进情况
3. Skill 反馈积分、等级、成就和未完成任务

### 5.2 奖励流程
1. 用户查看奖励列表
2. 用户提出兑换某个奖励
3. Skill 检查积分是否足够
4. 二次确认
5. 扣积分并记录兑换历史

## 7. 功能需求

### 6.1 早安播报

#### 触发方式
- 用户输入：`早安` / `开启今天` / `今日副本`
- 或定时触发

#### 早起判定口径
V1 中的“早起”定义为：

**用户在配置的晨间签到窗口内完成一次有效晨间签到。**

这里的“早起”不直接等于真实生理起床时间，而是 Skill 可判定的晨间行为信号。

有效晨间签到包括：
- 用户主动说“早安”
- 用户主动说“开启今天”
- 用户主动触发“今日副本”
- 宿主传入明确的晨间签到事件

默认配置建议：
- `morning_target_time = 07:00`
- `early_bird_grace_minutes = 30`

当签到时间早于 `07:30` 时，视为一次早起成功。

#### 输出内容
必须包含：
- 用户称呼
- 今日日期
- 累计生存天数
- 连续记录（如连续早起）
- 今日待办任务概览
- 当前积分与等级
- 一句鼓励语

#### 示例
```text
玩家DemoUser，早上好。
恭喜您成功通过昨日生存关卡。

当前生存时长：42天
连续早起：7天

【今日主线】
- 完成项目报告 | 80积分

【今日支线】
- 晨跑 | 20积分
- 阅读30分钟 | 20积分

当前积分：850
当前称号：资深玩家

今日副本已开启，祝您顺利通关。
```

### 6.2 任务系统

#### 任务类型
v1 仅保留两类任务：

##### 主线任务
- 重要且明确的任务
- 通常有截止时间
- 示例：完成项目报告、准备面试、提交方案

##### 支线任务
- 日常习惯或轻量目标
- 可每日重复
- 示例：晨跑、阅读30分钟、背单词

#### 说明
v1 不单独设计“长期任务”数据结构。长期坚持通过：
- 支线任务连续记录
- 成就系统
- 生存/早起 streak  
来承接。

---

### 6.3 创建任务

#### 触发条件
当用户表达未来意图时，Agent 应识别为创建任务。

#### 输入示例
- 我今天要完成项目报告
- 明天要准备面试
- 以后每天阅读30分钟
- 我想坚持晨跑

#### 输出要求
Skill 返回：
- 任务名称
- 任务类型
- 预计积分
- 截止信息（若识别到）

---

### 6.4 完成任务

#### 触发条件
当用户明确表达已完成行为时，Agent 应识别为完成任务。

#### 输入示例
- 我完成了项目报告
- 晨跑打卡了
- 今天阅读结束了

#### 输出要求
Skill 返回：
- 完成任务名称
- 获得积分
- 当前积分
- 是否解锁成就

---

### 6.5 积分系统

#### 规则
- 主线任务完成：50-100 分
- 支线任务完成：10-30 分
- 成就解锁：10-200 分

#### 原则
- v1 以加分为主
- 默认不做超时扣分
- 惩罚机制后置到未来版本

---

### 6.6 等级系统

#### 目标
给积分提供长期成长反馈，不做复杂数值设计。

#### 等级示例
- Lv1 新手玩家
- Lv2 资深玩家
- Lv3 高级玩家
- Lv4 精英玩家
- Lv5 大师玩家
- Lv6 传奇玩家

#### 展示要求
- 当前等级
- 当前称号
- 距离下一等级还差多少积分

---

### 6.7 成就系统

#### v1 成就范围

##### 生存类
- first_day
- survivor_7
- survivor_30

##### 早起类
- early_bird_3
- early_bird_7
- early_bird_30

说明：
早起类成就基于“晨间签到窗口内成功签到”的连续记录计算，而非用户真实起床时间。

##### 任务类
- task_master_10
- task_master_50

#### 解锁反馈
成就解锁时必须包含：
- 图标
- 名称
- 解锁原因
- 奖励积分

---

### 6.8 晚间结算

#### 触发方式
- 用户输入：`今天做了什么` / `每日结算` / `今天结算`
- 或定时触发

#### 输出内容
必须包含：
- 今日主线完成情况
- 今日支线完成情况
- 今日积分变化
- 今日新成就
- 当前等级
- 明日延续任务

#### 示例
```text
【地球Online 每日结算】
玩家DemoUser - 2026年3月24日

【今日完成】
- 主线任务：1/2
- 支线任务：1/2
- 今日获得积分：120

【任务详情】
✅ 面试准备
✅ 晨跑
⏳ 项目报告
❌ 阅读30分钟

【今日成就】
🌅 早起战士

【当前状态】
积分：950
等级：资深玩家
距离下一级还差：50分

明日副本仍包含：
- 完成项目报告
```

### 6.9 奖励兑换

#### 用户能力
- 查看奖励列表
- 兑换奖励
- 查看兑换历史

#### 奖励示例

## 8. 数据与状态原则

### 7.1 单一数据真相源
- 任务状态只在任务数据中维护
- 积分与等级只在积分数据中维护
- 成就只在成就数据中维护
- 奖励与兑换历史只在奖励数据中维护

### 7.2 数据闭环
任务完成时必须触发：
1. 更新任务状态
2. 更新积分
3. 检查成就
4. 写入历史
5. 将结果返回给 Agent

### 7.3 Agent 与脚本职责划分
#### Agent 负责
- 理解自然语言
- 判断用户意图
- 生成世界观文案
- 调用结构化工具
- 从宿主 memory / profile / recent context 整理 `host_context`

#### 脚本负责
- 更新数据
- 执行计算
- 返回结构化结果

## 10. 风险与应对

### 风险1：任务识别误判
应对：
- 由 Agent 做语义判断
- 必要时二次确认
- 支持撤销上一条

### 风险2：游戏化疲劳
应对：
- 不要每条都用重度 RPG 文案
- 关键节点强化仪式感，平时保持自然

### 风险3：功能边界失控
应对：
- v1 只守住任务闭环
- 生活数值化内容后置

### 风险4：数据不一致
应对：
- 统一数据入口
- 所有奖励/积分/成就更新走同一流程

### 风险5：平台耦合过深
应对：
- 不把 Skill 绑定到某个平台私有 memory 文件格式
- 统一使用 `host_context` 作为宿主到 Skill 的接入协议
- 通过 Host Adapter 做平台差异转换

## 11. 验收标准

v1 完成后需通过以下场景：

1. 用户说“早安”，能收到完整今日开局信息
2. 用户说“我今天要完成项目报告”，能正确创建主线任务
3. 用户说“以后每天阅读30分钟”，能正确创建支线任务
4. 用户说“我完成了项目报告”，能正确更新状态并加分
5. 连续记录或累计任务数达到阈值时，能解锁成就
6. 用户说“今天做了什么”，能输出每日结算
7. 用户说“我想兑换看电影”，能检查积分、确认兑换、记录历史
8. 在不同宿主平台下，Skill 仍能通过统一 `host_context` 获取必要上下文
