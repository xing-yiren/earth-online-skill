# Jiuwenclaw Dialogue Test Prompts

> 面向 jiuwenclaw 宿主的对话式测试文档。用于在真实 Agent 对话中验证地球Online Skill 是否成功接入，而不是只验证 Python 层逻辑。

---

## 使用说明

本测试文档用于验证：

- Skill 是否能在对话中被正确触发
- 宿主是否能把用户意图映射到 Skill 工具链
- runtime / onboarding / adapter / 任务 / 结算 / 奖励 是否真正连通

建议使用方式：

1. 在 jiuwenclaw 中打开一个新的对话会话
2. 按本文件中的顺序逐条输入 prompt
3. 记录每一条的实际返回结果
4. 将结果标记为：
   - 通过
   - 部分通过
   - 不通过

建议记录模板：

```markdown
### 用例编号
- 实际输入：
- 实际输出：
- 结果判断：通过 / 部分通过 / 不通过
- 备注：
```

---

## 一、初始化 / Onboarding

### 初始化验证总目标

当前建议先优先验证两类初始化场景：

1. **宿主缺少用户关键信息时**，是否会继续询问玩家姓名 / 时区，而不是直接完成初始化
2. **宿主已有用户关键信息时**，是否能正确读取并减少重复提问

建议优先先跑下面两个 A/B 场景。

---

### Case A1: 宿主缺少用户信息时启动

目标：
- 验证 `required_fields` 和 `next_action = ask_required_fields` 是否在真实宿主对话里生效

推荐 prompt：

```text
启动地球Online系统
```

预期结果：
- Agent 不应直接完成初始化
- 应继续询问至少以下字段中的缺失项：
  - 玩家称呼
  - 时区
- 如果信息不足，不应直接把玩家名落成 `玩家`

通过标准：
- 返回中出现“我应该怎么称呼你”或等价问题
- 返回中出现“你的时区是什么”或等价问题
- 没有直接跳过 onboarding 完成初始化

---

### Case A2: 宿主已有用户信息时启动

目标：
- 验证 adapter 是否能从宿主已有信息中读取用户画像
- 验证 onboarding 是否会减少不必要的重复提问

前提：
- 宿主侧 `USER.md` / memory 中已有可用用户信息

推荐 prompt：

```text
启动地球Online系统
```

预期结果：
- 如果已有名字 / 时区可信，Agent 可减少或跳过对这些字段的提问
- 最终欢迎信息应优先使用已知用户信息

通过标准：
- 最终欢迎页中玩家称呼不是默认值 `玩家`
- 没有把 Agent 名误当成用户称呼
- 如果字段已可信存在，不会重复追问同样问题

### Case 1: 首次启用地球Online

目标：
- 验证 Skill 是否进入初始化流程，而不是直接进入正常玩法

推荐 prompt：

```text
我想开始使用地球Online
```

预期结果：
- Agent 识别这是首次启用或初始化意图
- 如果当前尚未初始化，应开始 onboarding
- 不应直接跳到晨报或任务结算

通过标准：
- 返回中出现初始化、设置、确认信息，而不是直接进入正常业务流程

---

### Case 2: 确认称呼

目标：
- 验证 onboarding 能接住“我应该怎么称呼你”这类问题

推荐 prompt：

```text
你可以叫我小林
```

预期结果：
- Agent 识别这是 onboarding 配置回答
- Skill 后续应使用这个称呼

通过标准：
- 后续晨报或结算中出现 `小林` 或对应称呼

---

### Case 3: 确认晨间目标时间

目标：
- 验证晨间目标时间可通过对话完成初始化确认

推荐 prompt：

```text
默认早上七点算我的晨间目标时间，宽限三十分钟
```

预期结果：
- Agent 识别并完成 onboarding 配置更新
- 后续早起判定基于 07:00 + 30 分钟窗口

通过标准：
- 初始化完成后不再反复询问晨间目标时间

---

### Case 4: 初始化完成确认

目标：
- 验证 onboarding 在完成后能进入正常使用状态

推荐 prompt：

```text
现在初始化完成了吗？
```

预期结果：
- Agent 能明确表示已完成初始化，或者直接进入正常 Skill 使用状态

通过标准：
- 后续再触发业务时，不再重复走首次初始化流程

---

## 二、晨间签到与晨报

### Case 5: 晨间签到

目标：
- 验证“早安”能触发晨间签到链路

推荐 prompt：

```text
早安
```

预期结果：
- 触发晨报或今日开局信息
- 如果时间处于晨间窗口，应记录一次早起成功

通过标准：
- 返回中包含今日主线/支线、积分、等级或类似开局信息

---

### Case 6: 晨间重复签到

目标：
- 验证同日重复“早安”不会重复累计早起 streak

推荐 prompt：

```text
我再说一次，早安
```

预期结果：
- 系统不重复计算当天早起
- 仍可返回晨报，但不应重复发放同日签到收益

通过标准：
- 不出现重复新增 streak 或重复成就奖励迹象

---

### Case 7: 查询今天任务

目标：
- 验证晨报 / 今日副本查询可正常工作

推荐 prompt：

```text
今天有什么任务
```

预期结果：
- 返回当前 active 主线 / 支线任务

通过标准：
- 任务列表内容与之前创建的任务一致

---

## 三、任务创建

### Case 8: 创建主线任务

目标：
- 验证“未来意图”表达会被识别为主线任务创建

推荐 prompt：

```text
我今天要完成项目报告
```

预期结果：
- 创建一个主线任务
- 返回任务名称、任务类型、积分或截止时间相关信息

通过标准：
- 后续晨报或结算中能看到该任务

---

### Case 9: 创建支线任务

目标：
- 验证习惯型表达会被识别为支线任务

推荐 prompt：

```text
我想坚持每天阅读三十分钟
```

预期结果：
- 创建一个支线 daily 任务

通过标准：
- 后续晨报或结算中能看到 daily 支线任务

---

### Case 10: 重复创建支线任务

目标：
- 验证重复 daily 创建会被拦截

推荐 prompt：

```text
我还是想坚持每天阅读三十分钟
```

预期结果：
- 不应再创建一条重复的 daily 任务
- 应提示已存在相似任务或无需重复创建

通过标准：
- 不出现两条完全重复的 daily 任务

---

## 四、任务完成

### Case 11: 完成主线任务

目标：
- 验证完整任务完成闭环

推荐 prompt：

```text
我完成了项目报告
```

预期结果：
- 识别为完成任务
- 返回加分反馈
- 若达到阈值，可能触发成就

通过标准：
- 后续结算中该任务显示为已完成
- 积分发生变化

---

### Case 12: 再次完成同一个一次性任务

目标：
- 验证 once 任务不会被重复完成

推荐 prompt：

```text
项目报告我又完成了一次
```

预期结果：
- 不应重复加分
- 应提示任务已完成或无需重复登记

通过标准：
- 积分不再继续上涨

---

### Case 13: 完成支线任务

目标：
- 验证 daily 支线打卡逻辑

推荐 prompt：

```text
今天的阅读打卡完成了
```

预期结果：
- 识别为 daily 支线任务完成
- 返回支线积分反馈

通过标准：
- 今日结算能看到这条支线完成记录

---

### Case 14: 同日重复打卡 daily

目标：
- 验证 daily 当天不会重复计分

推荐 prompt：

```text
我再补一句，今天阅读也完成了
```

预期结果：
- 不应重复增加积分
- 应提示今日已完成

通过标准：
- 今日支线积分不重复累加

---

## 五、每日结算

### Case 15: 查询每日结算

目标：
- 验证结算链路是否工作

推荐 prompt：

```text
今天做了什么
```

预期结果：
- 返回今日完成情况
- 返回今日积分变化
- 返回今日成就
- 返回仍待推进任务

通过标准：
- 结算内容与当天实际对话行为基本一致

---

### Case 16: 重复触发结算

目标：
- 验证结算查询是只读行为，不会重复发奖励

推荐 prompt：

```text
再给我结算一次今天
```

预期结果：
- 可以再次展示结算
- 不应重复增加积分或重复发放奖励

通过标准：
- 重复结算前后状态不发生额外变化

---

## 六、奖励系统

### Case 17: 查看奖励列表

目标：
- 验证奖励列表可通过对话查询

推荐 prompt：

```text
有什么奖励可以换
```

预期结果：
- 返回奖励目录

通过标准：
- 列表中包含当前 seed/runtime 中可用奖励

---

### Case 18: 发起奖励兑换

目标：
- 验证奖励兑换前确认链路

推荐 prompt：

```text
我想兑换周末看电影
```

预期结果：
- 系统不应直接扣分
- 应先确认

通过标准：
- 回复中出现确认语义，而不是直接完成兑换

---

### Case 19: 确认兑换

目标：
- 验证正式兑换链路

推荐 prompt：

```text
确认兑换
```

预期结果：
- 扣积分
- 返回兑换成功反馈

通过标准：
- 后续结算或积分状态可反映扣分结果

---

### Case 20: 积分不足时兑换失败

目标：
- 验证余额不足场景

推荐 prompt：

```text
我想兑换休息一天
```

预期结果：
- 如果积分不足，应明确失败原因

通过标准：
- 不会错误扣分

---

## 七、建议测试顺序

建议按以下顺序在 jiuwenclaw 中逐条测试：

1. Case 1-4：初始化
2. Case 5-7：晨间签到与晨报
3. Case 8-10：任务创建
4. Case 11-14：任务完成
5. Case 15-16：每日结算
6. Case 17-20：奖励系统

---

## 八、对接成功的判断标准

地球Online 在 jiuwenclaw 上可认为“初步对接成功”，至少需要满足：

1. 首次对话能进入 onboarding，而不是直接跳过初始化
2. 晨间签到能触发晨报或开局信息
3. 创建任务后，后续查询能看到该任务
4. 完成任务后，积分和结算结果真正变化
5. 每日结算能反映当天对话中的真实推进情况
6. 奖励兑换前会确认，确认后才真正扣分
7. 宿主缺少用户信息时，初始化会继续追问而不是直接落默认值
8. 宿主已有用户信息时，初始化能正确读取并减少重复提问
